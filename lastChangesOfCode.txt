1. Обход rate-limit через подмену X-Forwarded-For (средняя серьёзность)
   Файл: core/view_utils.py (client_ip)
   Было: брался ПЕРВЫЙ элемент X-Forwarded-For — клиент мог подделать IP,

   Стало: порядок доверия X-Real-IP -> последний элемент XFF -> REMOTE_ADDR.
   Nginx (deploy-vps.sh) перезаписывает X-Real-IP и дописывает реальный IP
   в конец XFF, поэтому только эти значения достоверны.
   Тесты: ClientIpTests (3 шт.).

2. Неограниченный qty в корзине -> 500 на checkout
   Файл: core/cart_utils.py
   Было: qty любой величины; при создании заказа PositiveIntegerField
   (Order.total_cents / OrderItem.quantity) переполнялся на уровне БД.
   Стало: MAX_QTY_PER_ITEM = 999, клампится в add_item и set_qty.
   Тест: test_quantities_are_capped_per_item.

3. Logout по GET (forced-logout CSRF)
   Файл: core/views/auth.py (logout_view)
   Было: GET-запрос разлогинивал — чужой сайт мог выкидывать пользователей
   через <img src="/logout/">.
   Стало: POST-only (как LogoutView в Django 5). Все шаблоны и так шлют
   POST-форму. GET теперь возвращает 405, сессия не трогается.
   Тесты: test_logout_redirects_and_clears_session (POST),
   test_logout_rejects_get.

4. Конфликт EMAIL_USE_SSL + EMAIL_USE_TLS
   Файл: creativesphere/settings.py
   Было: EMAIL_USE_TLS по умолчанию включён; при EMAIL_USE_SSL=1 (Яндекс,
   порт 465) без явного TLS=0 Django падал при отправке письма.
   Стало: SSL автоматически отключает TLS (они взаимоисключающие).

Как накатить ЭТО обновление на VPS
==================================

Использовать, как и раньше, update-safe.sh. Два момента именно для
этого деплоя:

1. Сначала закоммитить и запушить изменения с рабочей машины —
   staged-удаления staticfiles + правки скриптов пока только локально.

2. Pull может ОДИН раз упасть на конфликте staticfiles/ (на сервере ещё
   старый update.sh без переходного guard'а, а collectstatic мог изменить
   отслеживаемые файлы). Если в логе "local changes would be overwritten":

     cd /var/www/Django
     git checkout -- staticfiles/
     sudo bash scripts/update-safe.sh

   Повторный запуск пройдёт чисто; все следующие обновления — обычный
   update-safe.sh без ручных шагов.

3. Проверить, что в .env на VPS есть строка DJANGO_DATABASE=postgresql —
   новый update-safe.sh определяет тип БД по ней. Если строки нет, скрипт
   решит, что БД SQLite, не найдёт файл и остановится (безопасно, но
   деплой прервётся).


Новый скрипт: scripts/ssl-renew.sh — 12.06.2026
===============================================

Продление Let's Encrypt сертификата + reload Nginx.

Основное авто-продление и так делает системный certbot.timer (ставится
пакетом certbot при деплое). ssl-renew.sh — страховка и ручной инструмент:

  * пишет даты истечения сертификатов в deploy/ssl-renew.log;
  * запускает certbot renew (трогает только сертификаты, у которых
    осталось < 30 дней — безопасно запускать хоть каждый день);
  * перезагружает Nginx ТОЛЬКО если сертификат реально продлился
    (--deploy-hook);
  * предупреждает, если certbot.timer выключен (и как его включить);
  * поддерживает --dry-run (репетиция без реального продления).

Запуск на VPS:
  sudo bash scripts/ssl-renew.sh                # обычный
  sudo bash scripts/ssl-renew.sh --dry-run      # проверка без продления

Cron-страховка (раз в неделю) — закомментированная строка добавлена в
scripts/crontab-daily.example.
